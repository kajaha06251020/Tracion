# Tracion MVP1 Design Specification

**Date:** 2026-03-15
**Status:** Draft
**Scope:** Web dashboard, multi-user auth, DAG graph visualization

---

## Overview

MVP1 is the first user-facing release of Tracion. Phase 0 established all backend infrastructure (API, SDKs, MCP server). MVP1 delivers the web dashboard that makes that data visible and useful.

**Target users:** Individual AI developers and small teams building with Claude Code, Cursor, or other AI coding tools.

**Core Aha moment:** A developer runs their AI agent, opens Tracion, and sees exactly what their agent did — every LLM call, every tool use, the causal graph of decisions — in a clean visual interface.

---

## Goals

1. Trace list and detail view with span waterfall visualization
2. Multi-agent causal graph (DAG) powered by React Flow
3. OAuth authentication (GitHub + Google) for multi-user access

**Out of scope for MVP1:**
- Real-time SSE notifications
- Cost analytics dashboard
- Billing or usage limits
- Accessibility audits
- Performance testing

---

## Architecture

```
[Better Auth in Hono API]
         │
         ▼
[apps/api: Hono + tRPC]  ←── HTTP/tRPC ───  [apps/web: Next.js 15]
         │                                            │
   [PostgreSQL]                          Pages:
   auth.users                            /login
   auth.sessions                         /traces
   auth.accounts                         /traces/[id]
   otel.traces                             /traces/[id]/graph
   otel.spans
```

### Auth placement

Better Auth runs inside `apps/api` using the Hono adapter. All OAuth callbacks, session creation, and session validation happen in the API process. The web app calls `/api/auth/*` endpoints for auth operations and reads session cookies on subsequent requests.

Benefits of this placement:
- Auth logic colocated with data (same PostgreSQL instance)
- No separate auth service to maintain
- Consistent with the project's self-hosted, privacy-first philosophy
- Leaves open the path to future API key authentication alongside session auth

### Session storage

Better Auth manages three tables automatically via migration:
- `auth.users` — user accounts (id, email, name, image)
- `auth.sessions` — active sessions (token, userId, expiresAt)
- `auth.accounts` — OAuth account links (provider, providerAccountId, userId)

Sessions are stored in PostgreSQL using the existing Drizzle connection. No Redis dependency added for MVP1.

## Pages

### `/login`

Single-purpose page with two OAuth buttons: GitHub and Google. No email/password form. On successful OAuth callback, redirects to `/traces`.

- Unauthenticated users are redirected here by Next.js middleware
- Authenticated users visiting `/login` are redirected to `/traces`

### `/traces` — Trace List

Infinite scroll table of all traces in the database.

**Data scoping (MVP1 — single-tenant):** MVP1 has no organization or user-level data isolation. All authenticated users see all traces. Multi-tenancy is deferred to a future milestone. This is consistent with the self-hosted, single-team deployment model.

**Columns:**

| Column | Content |
|--------|---------|
| Name | Trace name, links to `/traces/[id]` |
| Agent | agentId attribute |
| Status | `running` / `success` / `error` badge |
| Duration | Total wall-clock time (ms or s) |
| Tokens | Total token count across all spans |
| Cost | Total cost in USD |
| Time | Relative time ("2 min ago") |

**Filters (top bar):**
- Status: all / running / success / error
- Agent ID: free-text input
- Date range: preset options (last hour, last 24h, last 7d) + custom

**Pagination:** Cursor-based via existing `traces.list()` tRPC procedure. Infinite scroll using Intersection Observer.

**Aggregated fields:** Trace-level `totalTokens` = sum of (`inputTokens + outputTokens`) across all spans. `totalCostUsd` = sum of `costUsd` across all spans. These are pre-computed at ingest time and stored on the `otel.traces` row.

### `/traces/[id]` — Trace Detail

Two-panel layout:

**Left panel: Span waterfall**
- Gantt-style visualization ordered by `startTime ASC, id ASC` (ULID encodes insertion time, so ties break naturally by insertion order)
- Each row: indent level (= depth in `parentSpanId` tree, root = 0), span name, span kind chip, duration bar
- Color of the duration bar maps to span kind: llm=blue, tool=green, agent=purple, retrieval=orange, custom=gray
- Click a row to populate the right panel

**Right panel: Span attributes**
- Appears on span selection
- Shows: model, input tokens, output tokens, cost, status, events list
- Raw attributes expandable as JSON

**Header:**
- Trace name, total duration, total tokens, total cost, status badge
- Tab navigation: "Waterfall" (current page) | "Graph" (links to `/traces/[id]/graph`)

### `/traces/[id]/graph` — DAG Graph

React Flow canvas showing the causal graph of all spans in the trace.

**Nodes:**
- One node per span
- Label: span name
- Color by span kind (same palette as waterfall)
- Click to open span detail sidebar (same attributes panel as detail page)

**Edges:**
- Derived from `parentSpanId` relationships
- Directed arrows from parent to child
- No additional API call required — edges are computed client-side from `spans.listByTrace()`

**Layout:**
- Dagre auto-layout (top-down)
- Fit-to-screen on initial load
- Zoom and pan controls

**Tab navigation:** "Waterfall" | "Graph" (active)

## Data Flow

### Auth flow

```
User clicks "Sign in with GitHub"
  → GET /api/auth/signin/github  (Better Auth, in Hono API)
  → GitHub OAuth redirect
  → GET /api/auth/callback/github
  → Better Auth creates/updates auth.users + auth.accounts
  → Sets session cookie
  → Redirect to /traces
```

### Trace list flow

```
/traces page mounts
  → useInfiniteQuery via tRPC client
  → POST /trpc/traces.list (with session cookie)
  → Hono middleware validates session via Better Auth
  → traces.list() service executes DB query
  → Returns paginated results
  → Intersection Observer triggers next page fetch
```

### DAG graph flow

```
/traces/[id]/graph page mounts
  → useQuery via tRPC: spans.listByTrace(traceId)
  → Returns flat array of spans with parentSpanId
  → Client-side transform: spans → { nodes: [], edges: [] }
  → React Flow renders with dagre layout
```

The DAG transform is a pure function:

```ts
function spansToGraph(spans: Span[]): { nodes: Node[]; edges: Edge[] } {
  const nodes = spans.map(span => ({
    id: span.id,
    data: { label: span.name, kind: span.kind },
    position: { x: 0, y: 0 }, // overridden by dagre
  }));
  const edges = spans
    .filter(span => span.parentSpanId)
    .map(span => ({
      id: `${span.parentSpanId}-${span.id}`,
      source: span.parentSpanId!,
      target: span.id,
    }));
  return { nodes, edges };
}
```

## API Changes

### New: Auth middleware

`requireSession` is added to web-facing tRPC routes only. The existing `POST /v1/traces` ingest endpoint retains its current `optionalApiKey` middleware — SDK and MCP clients use API key auth and cannot hold browser session cookies.

```ts
// apps/api/src/middleware/auth.ts
export const requireSession = createMiddleware(async (c, next) => {
  const session = await auth.api.getSession({ headers: c.req.raw.headers });
  if (!session) return c.json({ error: 'Unauthorized' }, 401);
  c.set('user', session.user);
  await next();
});
```

### New: tRPC + Hono wiring

The tRPC Hono route is wrapped with `requireSession`. The validated user is forwarded into the tRPC context factory:

```ts
// apps/api/src/index.ts
app.use('/trpc/*', requireSession);
app.use('/trpc/*', trpcServer({
  router,
  createContext: (c) => ({ user: c.get('user'), db }),
}));
```

The tRPC context type becomes:

```ts
type Context = {
  user: User;  // populated by requireSession
  db: Database;
};
```

No new tRPC procedures are required for MVP1 — all data operations use existing routes.

**Design constraint:** All tRPC routes are browser-only by design. SDK and MCP clients must use the REST ingest endpoint (`POST /v1/traces`) and must not call tRPC directly. If SDK-facing tRPC procedures are ever needed in the future, they must live under a separate path (e.g., `/trpc-internal/*`) that excludes `requireSession`.

### Existing tRPC routes (no changes needed)

- `traces.list()` — already supports agentId, status, cursor pagination
- `traces.get(id)` — returns full trace
- `spans.listByTrace(traceId)` — returns all spans for DAG + waterfall
- `spans.get(id)` — single span attributes

## Testing Strategy

### Unit tests (Vitest)

- `spansToGraph()` transform function — pure function, tests cover: linear chain, branching, root-only. Cycles are invalid by contract (OTel spans cannot have circular parent references) and are not handled.
- Better Auth session middleware — mock Better Auth, assert 401 on missing session, 200 on valid session
- tRPC auth guard — assert unauthenticated requests are rejected

### Integration tests (Vitest + testcontainers)

- Better Auth session creation with mocked OAuth token exchange (unit-level, no browser)
- Full request cycle: session cookie → tRPC → DB → response

### E2E tests (Playwright)

Three critical paths:

1. **Auth:** Visit `/traces` → redirect to `/login` → mock OAuth provider → callback → redirect to `/traces` → traces table visible (covers full OAuth redirect + cookie flow)
2. **Trace detail:** Click trace row → waterfall renders → click span → attributes panel shows model + tokens
3. **DAG graph:** Click "Graph" tab → React Flow canvas renders → nodes and edges visible

## Success Criteria

MVP1 is complete when:

- [ ] A new user can sign in with GitHub or Google
- [ ] Unauthenticated routes redirect to `/login`
- [ ] `/traces` shows a scrollable list of traces with filters
- [ ] `/traces/[id]` shows a span waterfall with clickable span attributes
- [ ] `/traces/[id]/graph` shows a React Flow DAG of all spans
- [ ] All unit and integration tests pass
- [ ] At least one Playwright E2E test covers the full auth → traces flow
- [ ] `docker compose up` brings up the full stack and a Playwright E2E test passes against it: SDK sends a trace → trace appears in `/traces` → detail and graph pages render correctly
